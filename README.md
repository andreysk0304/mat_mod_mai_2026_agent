# AI-агент для анализа репозитория и планирования работ

Трек: A+D

Консольный агент на Python, который использует LLM и реальные инструменты, чтобы отвечать не только текстом, но и данными из внешнего API и результатом оптимизационной модели.

Основной сценарий: пользователь просит оценить GitHub-репозиторий или распределить ограниченный бюджет времени между задачами. Агент сам решает, когда вызвать инструмент, получает структурированный результат и объясняет решение обычным языком.

## Что закрывает проект

**Трек A: агент + API**

- инструмент `github_repo_info` ходит в GitHub REST API;
- получает описание репозитория, звезды, форки, открытые задачи, даты активности, лицензию и языки;
- поддерживает публичные репозитории без токена и `GITHUB_TOKEN` для большего лимита.

**Трек D: агент для оптимизации**

- инструмент `plan_optimizer` принимает задачи, часы и ценность;
- считает базовый вариант: жадный выбор по `value_per_hour`;
- считает улучшенный вариант: задачу о рюкзаке 0/1 через динамическое программирование;
- возвращает сравнение базового и улучшенного вариантов, а также список задач, которые не вошли в план.

## Быстрый запуск

```bash
python3 main.py
```

Перед первым запуском создайте `.env` из шаблона и укажите ключ LLM:

```bash
cp .env.example .env
```

Минимально нужны:

```env
AI_AGENT_API_KEY=...
AI_AGENT_MODEL=gpt-4o-mini
AI_AGENT_BASE_URL=https://api.openai.com/v1
```

Для GitHub API можно добавить:

```env
GITHUB_TOKEN=...
```

Токен необязателен для публичных репозиториев, но помогает не упереться в лимит запросов.

## Примеры для скринкаста

### Сценарий 1: GitHub API

```text
Проанализируй репозиторий openai/openai-python: насколько он активен, какие языки использует и какие риски видны по открытым issues?
```

Ожидаемо агент вызовет:

```text
[tools: github_repo_info]
```

### Сценарий 2: оптимизация плана

```text
У меня есть 8 часов на мини-хак. Задачи:
- README 1ч, ценность 3
- инструмент GitHub API 4ч, ценность 10
- инструмент оптимизации 4ч, ценность 10
- REFLECTION 1ч, ценность 3
- скринкаст 3ч, ценность 8
- полировка консольного интерфейса 1ч, ценность 3
- тесты 3ч, ценность 6

Составь план, сравни базовый и улучшенный варианты, объясни что не делать.
```

Ожидаемо агент вызовет:

```text
[tools: plan_optimizer]
```

### Сценарий 3: комбинированный A+D

```text
Посмотри GitHub-репозиторий pallets/flask и предложи план из 8 часов для подготовки учебного демо: что показать, что проверить и что отложить. Сравни простой базовый вариант и оптимизированный план.
```

В этом сценарии агент может сначала вызвать `github_repo_info`, а затем `plan_optimizer`.

## Доступные инструменты

Предметные инструменты для сдачи:

- `github_repo_info` - получает данные о GitHub-репозитории через внешний API;
- `plan_optimizer` - решает задачу распределения времени и сравнивает базовый вариант с улучшенным решением.

В проекте также остались служебные инструменты агента:

- `calculate` - безопасно считает арифметические выражения;
- `get_current_time` - возвращает локальное время процесса;
- `ls` - показывает файлы внутри проекта;
- `mkdir` - создает директории внутри проекта;
- `read_project_file` - читает текстовые файлы проекта;
- `write_project_file` - создает, перезаписывает или дописывает файлы внутри проекта.

Служебные инструменты не являются главным треком, но помогают показать, что агент работает через общий механизм вызова инструментов.

## Данные и источники

- GitHub API: `https://api.github.com/repos/{owner}/{repo}`
- GitHub languages API: `https://api.github.com/repos/{owner}/{repo}/languages`
- Пример входных задач: `data/example_tasks.json`
- Дата подготовки примера: 2026-04-24

Инструмент `github_repo_info` возвращает поле `collected_at`, чтобы в ответе было видно, когда были получены данные.

## Структура проекта

```text
main.py                 # входная точка консольного приложения
agent/                  # среда выполнения агента, совместимый с OpenAI клиент, память, реестр инструментов
tools/github_repo_info.py
tools/plan_optimizer.py
tools/                  # остальные служебные инструменты
data/example_tasks.json # пример задачи для оптимизатора
REFLECTION.md           # эссе по требованиям мини-ТЗ
README.md               # инструкция запуска и сценарии демо
```

## Команды консольного приложения

- `/help` - список команд;
- `/tools` - показать доступные инструменты;
- `/reset` - очистить историю диалога;
- `/reload` - перезагрузить инструменты и навыки после изменения файлов;
- `/log on` и `/log off` - включить или выключить логирование;
- `/exit` - выход.

После ответа агента консольное приложение печатает список реально использованных инструментов, например:

```text
[tools: github_repo_info, plan_optimizer]
```

Это удобно для проверки требования, что агент не просто вызывает LLM, а действительно использует инструменты.

Ответы агента выводятся как Markdown: заголовки, **жирный текст**, встроенный код и таблицы в терминале форматируются в более читаемый вид без дополнительных зависимостей.

## Что использовано

Основа проекта - собственная минимальная среда выполнения на Python:

- совместимый с OpenAI `/chat/completions`;
- вызовы функций и инструментов;
- динамическая загрузка инструментов из папки `tools/`;
- память текущей консольной сессии.

Внешние фреймворки для агентов не используются. Оптимизация реализована самостоятельно в `tools/plan_optimizer.py`.
