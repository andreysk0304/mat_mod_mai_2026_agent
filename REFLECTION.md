# Reflection

## Почему эта задача

Я выбрал идею агента-планировщика для разработки, потому что она хорошо совпадает с реальной ситуацией мини-хака: времени мало, требований много, а решение надо не просто придумать, но и довести до демонстрации. Мне было интересно сделать не очередной чат, который красиво рассуждает, а маленького помощника, который умеет брать внешние факты о репозитории и превращать их в конкретный план работ. Такой агент полезен студенту, начинающему разработчику или участнику хакатона: он помогает быстро понять, насколько живой проект, какие части стоит показать в демо и как распределить ограниченный бюджет времени.

В проекте уже был CLI-агент с поддержкой OpenAI-compatible API, tools, skills и памяти диалога. Поэтому я не стал переписывать основу, а сфокусировался на двух инструментах, которые дают агенту предметную пользу. Первый инструмент ходит в GitHub API и получает данные о репозитории: описание, звезды, forks, открытые issues, даты обновления и языки. Второй инструмент решает задачу оптимизации времени: считает простой baseline и улучшенное решение через динамическое программирование. За счет этого агент может не только сказать “сделай сначала важное”, но и показать сравнение двух стратегий.

## Где агент сломался

Первый косяк связан с качеством входных данных. Если пользователь описывает задачи слишком расплывчато, например “доделать проект, записать видео, все проверить”, модели приходится самой придумывать часы и ценность задач. Оптимизатор честно решает математическую задачу, но результат становится слабым, потому что входные оценки взяты из воздуха. Это не баг алгоритма, а граница ответственности агента, ему нужны хотя бы примерные длительности и приоритеты.

Второй косяк связан с GitHub API. Для публичных репозиториев инструмент работает без токена, но быстро упирается в rate limit или не может открыть приватный репозиторий. Я добавил поддержку переменной `GITHUB_TOKEN`, но агент все равно должен явно объяснять пользователю, что часть данных недоступна, а не делать вид, что репозиторий пустой или неактивный.

## Что сделал бы дальше

Если бы было больше времени, я бы добавил отдельный режим `demo`, который запускает заранее подготовленный сценарий без ручного ввода в CLI. Это упростило бы скринкаст и проверку проекта. Еще я бы добавил тесты для обоих новых tools: парсинг GitHub URL, обработку ошибок API, проверку baseline и knapsack на маленьких примерах, где правильный ответ очевиден.

Следующий шаг по качеству агента - сделать уточняющие вопросы перед оптимизацией. Сейчас модель может сразу вызвать `plan_optimizer`, даже если не хватает оценок времени. Более надежное поведение: сначала спросить недостающие часы или приоритеты, затем уже решать задачу. Также можно сохранять снимки данных GitHub в `data/`, чтобы агент мог сравнивать репозиторий во времени: например, выросло ли число issues, менялся ли основной язык, насколько давно был последний push.
